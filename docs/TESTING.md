# Testing — what's actually verified, and what isn't

Run everything: `pip install -r requirements-dev.txt && python3 -m pytest tests/ -v`
(81 tests, currently all passing.)

This file exists because "I ran the app once and it didn't crash" is not
the same claim as "this is tested." Below is what each test file actually
checks, and — just as important — what none of this covers.

## What's covered

**`tests/test_data_integrity.py`** — is `data/network.json` itself even a
valid graph. Every edge endpoint and corridor reference points at a real
node, no duplicate IDs, no negative capacities, scenario probabilities sum
to 1.0, and (the one that would have caught a real class of bug) every
supplier with nonzero flow actually has an outbound edge — a supplier with
flow but no edges would be silently invisible to the optimizer via
`pyo.Constraint.Skip`, with no error anywhere.

**`tests/test_scenario_engine.py`** — `degraded_capacity()` at its
boundaries (severity 0 and 1, wrong corridor, non-shipping edge types,
severity>1 which the UI can't produce but a caller could), the incremental
-vs-baseline netting logic (baseline vs itself must be exactly zero,
incremental can never exceed raw), monotonicity (hormuz_100 must be at
least as bad as hormuz_50), and that `days_of_cover` degrades to `None`
rather than raising `ZeroDivisionError` when there's no SPR capacity.

**`tests/test_optimizer.py`** — this is the one that matters most, because
it's the only real math in the system. Checks actual constraint
satisfaction on the solved model (not just "solver said optimal"):
demand-balance holds to solver tolerance, contracts never exceed
undisrupted capacity, delivered flow never exceeds contract or surviving
capacity, take-or-pay floor is respected, supplier export caps hold, no
variable goes negative. Also re-asserts the two previously-diagnosed
failure modes as regression tests (risk-aversion must actually change the
solution; a shortfall penalty below deliverable cost must cause collapse)
so a future edit can't silently reintroduce either bug without a test
failing. **Found and fixed one real bug**: `run_full_network(scenario_keys=[])`
was silently substituting the full 4-scenario default instead of erroring,
because `scenario_keys or list(SCENARIOS.keys())` treats an empty list as
falsy in Python. Fixed to check `is None` explicitly.

**`tests/test_signal_extractor.py`** — every one of the 7 pre-cached
headlines runs through `extract_signal()` without raising, `nearest_scenario()`
boundary conditions (0.74 vs 0.75 severity is the hormuz_50/hormuz_100 cutoff),
and that malformed/partial input (missing keys, unknown corridor strings)
degrades to `baseline` rather than crashing.

**`tests/test_briefing_generator.py`** — the edge cases the happy-path demo
click never exercises: no headline (manual scenario pick), zero supplier
shifts, `days_of_cover=None` (baseline case), and a spot-check that every
number quoted in the briefing text actually traces back to the optimizer's
own output rather than being invented by the template.

**`tests/test_app_integration.py`** — `streamlit.testing.v1.AppTest`-driven
stress paths, not just the one happy path exercised during development:
all 7 headlines through the full pipeline (not just one), baseline scenario
through the Timeline's "nothing to recover from" branch, λ at both
extremes (0 and 6), the Day slider at both extremes (0 and day==n_days),
switching the scenario dropdown after a solve without re-clicking "Run
optimizer," all four Assumptions sliders pushed to both extremes
simultaneously, two consecutive headline extractions to check session
state doesn't leak between them, and the manual-only path that never
touches Layer 1 at all.

## What's NOT covered — disclosed, not hidden

- **No visual verification.** Nothing here can confirm the map actually
  *looks* right — path geometry, color choices, marker placement are all
  checked by reading the underlying data, never by rendering. The map/route
  fixes earlier in this build were verified this way out of necessity, not
  by choice.
- **No real LLM call tested.** `ANTHROPIC_API_KEY` isn't configured in this
  environment, so every test above exercises the rule-based/template
  fallback path. The `llm.py` → real API path is written defensively
  (catches all exceptions, returns `None` on any failure) but has never
  actually been exercised end-to-end against the real Anthropic API.
- **No load/concurrency testing.** This is a single-process Streamlit app
  for a live demo to a handful of judges, not a production service —
  concurrent users, session state collisions across browser tabs, or
  solver contention under load are out of scope and untested.
- **No adversarial data testing.** These tests assume `network.json` was
  generated by `build_network.py` as intended. Hand-corrupting the JSON
  (missing required fields, wrong types) isn't tested — the app would
  likely throw an unhandled `KeyError`/`TypeError` on load, not a graceful
  error message.
- **Numerical tolerance, not exact equality**, is used throughout for
  constraint checks (`1e-4` to `1e-6`), which is standard for LP solver
  output but means truly minuscule constraint violations wouldn't fail a
  test. This is appropriate for a linear program with continuous
  variables, not a shortcut taken to make tests pass.
